案例 53：pImpl 编译防火墙

使用 pImpl 惯用法隐藏实现细节，减少编译依赖，加速编译。

核心知识点：pImpl、前向声明、unique_ptr、编译隔离

完整代码（含逐行注释）

// 案例 53：pImpl 编译防火墙
// 隐藏实现细节，减少头文件依赖
#include <iostream>
#include <memory>
#include <string>
#include <vector>
using namespace std;

// === 头文件（public header）===
class Widget {
public:
    Widget();                           // 构造
    ~Widget();                          // 析构（需在 .cpp 中定义）
    Widget(Widget&&) noexcept;          // 移动
    Widget& operator=(Widget&&) noexcept;

    void setName(const string& name);
    string getName() const;
    void addData(int value);
    void print() const;

private:
    // 前向声明实现类（在 .cpp 中定义）
    struct Impl;
    unique_ptr<Impl> pImpl;             // 指向实现的指针
};

// === 实现文件（.cpp）===
struct Widget::Impl {
    string name;
    vector<int> data;

    void print() const {
        cout << "Widget: " << name << endl;
        cout << "Data: ";
        for (int v : data) cout << v << " ";
        cout << endl;
    }
};

// Widget 的成员函数都委托给 pImpl
Widget::Widget() : pImpl(make_unique<Impl>()) {
    cout << "Widget 构造" << endl;
}

// 析构函数必须在 Impl 完整定义之后
Widget::~Widget() = default;

Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

void Widget::setName(const string& name) {
    pImpl->name = name;
}

string Widget::getName() const {
    return pImpl->name;
}

void Widget::addData(int value) {
    pImpl->data.push_back(value);
}

void Widget::print() const {
    pImpl->print();
}

int main() {
    Widget w;
    w.setName("pImpl Widget");
    w.addData(10);
    w.addData(20);
    w.addData(30);
    w.print();

    // 移动
    Widget w2 = move(w);
    w2.print();

    return 0;
}

语法规则：pImpl 惯用法结构

// 头文件
class Widget {
    struct Impl;  // 前向声明
    unique_ptr<Impl> pImpl;
public:
    void doSomething();
};
// 实现文件
struct Widget::Impl { /* 真实的成员 */ };
Widget::Widget() : pImpl(make_unique<Impl>()) {}

举一反三：pImpl 将类的实现细节移到 .cpp 文件，用户看不到也不依赖它们。改变 Impl 不需要重新编译使用 Widget 的代码。是大型项目的核心编译优化技术。

语法规则：析构函数的特殊处理

// 头文件中: ~Widget();
// .cpp 中: Widget::~Widget() = default;
// 必须在 Impl 完整定义之后才能 default 析构函数

举一反三：unique_ptr 的析构需要完整类型定义才能 delete。如果在头文件中 = default 析构，编译器会在此处实例化 unique_ptr 的析构但 Impl 不完整，导致编译错误。

语法规则：编译防火墙的好处

// 1. 修改实现不影响接口用户（只需重新链接）
// 2. 头文件不暴露内部类型、库依赖
// 3. ABI 兼容：实现改变不破坏二进制兼容

举一反三：代价：多一层间接访问（轻微性能开销）、多一次动态分配、代码量增加。Bridge 模式是 pImpl 的设计模式对应物。
